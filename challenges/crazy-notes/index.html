Flujo del Programa

Este programa muestra un menu con las siguientes opciones:

  1. Alocar una estructura world de 0x18 bytes, y un buffer de tamaño entre 0x30 y 0x50 bytes.
  2. Editar una estructura world y su buffer, si esta en rango de heap.
  3. Mostrar los datos de una estructura world y despues elegir si liberar la estructura y el buffer. Al liberar eliminamos todos los datos de la estructura world, pero no perdemos la referencia a esta (¡UAF!).

Estrategia de Explotacion

Normalmente cuando hago retos de heap, mi metodología es la siguiente:

  1. Conseguir Arbitrary Read/Write o OOB Read/Write.
  2. Leakear Libc.
  3. Buscar forma de ganar Code Execution

Ganando Arbitrary Read & Write

Sólo tenemos 8 alocaciones, tenemos que usarlas sabiamente. Al crear un world se hacen dos malloc(), uno de 0x20 bytes del que controlamos totalmente los primeros 0xc bytes, y un segundo de un tamaño a elegir entre 0x30 y 0x50 que controlamos totalmente. La idea es, alocar tres world, liberar el primero y mostrar los datos para leakear la Heap Base, despues liberar el segundo y hacer un TCache Poisoning para alocar un chunk en el void* buffer del segundo. Con esto podremos controlar ese puntero y ganar Arbitrary Read & Write dentro del heap. El tercero es importante para más adelante.

Este reto usa la versión 2.39 de glibc, por lo que hay Safe Linking.

Safe Linking

El Safe Linking es una medida de seguridad de la TCache, lo que hace es en el Forward Pointer de los chunks liberados guarda (heap_base >> 12) XOR next_ptr en vez de next_ptr. Por lo que para hacer un TCache Poisoning tendremos que leakear la Heap Base primero. Esto es bastante sencillo con nuestro UAF, basta con leer el Forward Pointer del primer chunk liberado, ya que será (heap_base >> 12) XOR 0.

Ahora modificando los primeros 8 bytes en el chunk 0 controlamos el buffer del chunk1. Con esto hecho, ¿cómo leakeamos Libc?

Leakeando Libc

Normalmente en retos de heap se suele leakear Libc haciendo que un chunk caiga en la unsorted bin, ya que cuando un chunk cae en la unsorted el fd y bk apuntan a la main arena. Pero aquí no podemos alocar un chunk lo suficientemente grande para que al liberarlo vaya a la unsorted.

Lo que sí podemos hacer es crear un chunk falso en el heap, y poner su dirección en el buffer de un world, para que al liberar el world se libere también nuestro chunk falso y caiga en la unsorted bin.

Aquí hay varias cosas que tener en cuenta:

  1. Tiene que ser de un tamaño mayor a 0x410 para que no caiga en la TCache. (Yo lo haré de 0x500)
  2. Tenemos que crear un guard chunk falso despues para que no consolide.
  3. Para que el heap se crea que el guard chunk está en uso tenemos que crear otro chunk falso después con el PREV_INUSE bit puesto.

Ganando Code Execution

Ya tenemos nuestro leak de libc, ¿y ahora qué?

Una vez llegados a este punto hay varias formas para ganar Code Execution, una de ellas es con la tls_dtor list, os dejo el writeup del creador del reto usando esta técnica Writeup with TLS_dtor list

Yo resolveré el reto con FSOP. Crearé un fake struct _IO_FILE_plus en el heap y lo pondré en la _IO_list_all para que al ejecutarse _IO_flush_all() cuando salgamos del programa nos haga system("/bin/sh").

FSOP (File Stream Oriented Programming)

FSOP es una técnica de explotación en glibc que consiste en abusar de las estructuras internas de los streams de fichero (FILE, usadas por funciones como printf, fgets, etc.) para conseguir ejecución de código.

La idea es que glibc representa los ficheros mediante estructuras(_IO_FILE) que contienen punteros a funciones (vtable). Si un atacante consigue corromper una de estas estructuras (por ejemplo, mediante un heap overflow o un Use After Free), puede redirigir esos punteros para que, cuando el programa invoque una operación sobre el stream, acabe ejecutando código controlado.

Sobrescribiendo la vtable

Para ganar ejecución arbitraria de código habría que conseguir sobrescribir la vtable para que cuando _IO_flush_all() vaya a llamar a una funcion de nuestra File Struct realmente llame a system. Aquí teneis un diagrama para que podaís visualizarlo mejor.

source https://pwn.college/software-exploitation/file-struct-exploits/

Pero aquí nos encontramos con un problema, glibc tiene una protección en la que verifica que la vtable este en la sección __libc_IO_vtables.

Más información en: https://elixir.bootlin.com/glibc/glibc-2.31/source/libio/libioP.h#L935

Bypasseando la protección de la vtable

Para bypassear la protección podemos sobrescibir la vtable dentro de la sección __libc_IO_vtables para que llame a _IO_wfile_overflow, esta función luego llama a _IO_wdoallocbuf que usa una vtable sin protección en FILE->wide_data

Al hacer eso y sobrescribir la vtable en wide_data podremos llamar a system()

Ahora bien, ¿como le pasamos la cadena "/bin/sh"?, esto es relativamente sencillo, porque al llamar a cualquier función de la vtable, el puntero al File Struct se pasa como primer argumento. Solamente habría que poner al principio de la File Struct la cadena, ¿no?

Pues no podremos poner directamente "/bin/sh" porque al principio de la File Struct están las flags, y las corromperemos de tal manera que no se llamará a _IO_wfile_overflow. La solución es poner la cadena "\x3b\x01\x01\x01\x01\x01\x01\x01", con esta cadena pasaremos los checks de las flags, y como \x3b representa un ";" se ejecutara la siguiente cadena como otro comando.

También tenemos que tener otras cosas en cuenta:

  1. _IO_write_ptr tiene que ser mayor a 0 y menor que _IO_write_end
  2. _lock tiene que ser una dirección de memoria escribible y con valor 0.

Sobrescribir la _IO_list_all

Para sobrescribir la _IO_list_all no podemos usar el mismo método de Tcache poisoning de antes, porque superariamos el numéro maximo de worlds creados.

Como tenemos Arbitrary Write en el heap podemos sobrescribir la Tcache Entry y modificar la entry de chunks de 0x20 para que el proximo que devuelva sea la _IO_list_all. Una vez hecho eso simplemente escribimos el puntero a nuestra fake File Struct y salimos del programa.
