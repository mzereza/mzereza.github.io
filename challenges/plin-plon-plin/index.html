El programa consiste en dejar que el usuario introduzca dos veces una string de hasta 0x80 bytes y el programa la mostrara inseguramente con printf. Tambien tenemos una función print_flag que nos mostrará la flag.

La estrategia de explotacion será la siguiente:

  1. Leakear PIE con Format Strings.
  2. Sobrescribir la entrada GOT de puts con la dirección de print_flag.
  3. Al llamar a puts tendremos la flag.

Fase 1: Leakear

¿Qué es una Format String Vulnerability?

Una format string vulnerability aparece cuando funciones como printf interpretan directamente la entrada del usuario como formato (printf(user_input)), permitiendo usar especificadores como %p, %x, %s o %n. Esto da control sobre la lectura del stack (leaks de direcciones útiles para romper PIE/ASLR) e incluso escritura arbitraria en memoria mediante %n. En explotación real, esto permite construir primitivas de lectura/escritura sin necesidad de overflow, siendo ideal para atacar estructuras como la GOT.

En retos de format string vulnerability siempre me gusta hacer una función para que leeakee varios índices, y de ahí ya elijo cuales usar. La función se ve así:

from pwn import *

def leak_fmt(start, end):
    payload = b"ABCDEFG"
    for i in range(start, end):
        payload += f".%{i}$p".encode()
    p.sendline(payload)
    p.recvline()
    data = p.recvline()[:-1]
    parse = data.split(b".")
    for idx, value in enumerate(parse):
        log.info(f"{idx+start-1}: ".encode() + value)

Esta funcion tambien me sirve para ver el índice donde empieza mi input, donde veamos un 0x2e47464544434241 significa que en ese índice empieza nuestro input. Ese valor representa “ABCDEFG.” en Little Endian.

Leakeando los primeros 10 valores vemos que el inicio de nuestro input está en el indice 6.

Leakeando los siguientes 20 vemos que tenemos un leak del PIE en el indice 25.

Fase 2: Sobrescribir la GOT

Global Offset Table (GOT)

La Global Offset Table (GOT) es una tabla en memoria que almacena las direcciones reales de funciones externas (como puts, printf, etc.) una vez son resueltas en tiempo de ejecución. Cuando el programa llama a una función de libc, en realidad accede primero a su entrada en la GOT, que apunta a la dirección correcta. Si puedes sobrescribir una de sus entradas (por ejemplo, puts), puedes redirigir la ejecución hacia cualquier función, como print_flag.

El siguiente paso despues de leakear el PIE es sobrescribir la entrada de puts en la GOT. Para no complicarnos la vida usaremos la funcion fmtstr_payload de pwntools, esta función recibe como argumentos: fmtstr_payload(input_idx, {addr_to_overwrite: value}, write_size="byte|short|int"). El write_size se utiliza para que el payload se haga con %n, %hn o %hhn, lo que cambia el tamaño del output y varía su eficiencia.
