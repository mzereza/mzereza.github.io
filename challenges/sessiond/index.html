Flujo de programa:

El programa nos muestra un menú con distintas funciones:

La función manage nos pide un tamaño para introducir datos, después lee nuestros datos en un segmento de la .BSS y los copia a una variable en el stack. Aquí hay un buffer overflow, ya que podemos escribir 0x210 bytes, pero el buffer es de 0x200 bytes. !Tenemos la capacidad de sobrescribir el Saved RBP y el Saved return address!

La función login pone la direccion de memoria de input en el offset 0x4070 y nos pide un nombre de usuario, el cual lee con read en el offset 0x4060, esto será importante para mas adelante, y después lo muestra en pantalla.

La función status se ve rara en el pseudocodigo de Ghidra, así que vamos a mirar directamente el assembly:

En el assembly se ve que printea con la syscall write una string y hace un JMP al return, pero entre medias hay un gadget muy poderoso POP RAX; SYSCALL; RET. Este gadget nos permite ejecutar la syscall que queramos. A este gadget no podemos llegar de manera convencional, pero si conseguimos controlar el RIP podemos saltar a esa instrucción.

La pregunta aquí es: ¿Cómo podemos pasar de esto a ejecución arbitraria de comandos?

Podemos sobrescribir el Saved RIP con la dirección del gadget, pero no tenemos ningun control sobre los registros ni el resto del stack. Además, ¿que podemos hacer con una sola syscall?

Sigreturn Oriented Programming (SROP)

SROP es una técnica de explotación que aprovecha la syscall sigreturn para controlar completamente el estado de los registros de la CPU. En sistemas Linux, sigreturn se usa internamente para restaurar el contexto de un proceso tras manejar una señal, cargando los valores de los registros desde una estructura en el stack (sigcontext).

La clave es que, si un atacante puede forzar la ejecución de sigreturn (por ejemplo, mediante un gadget o una syscall), puede construir manualmente ese contexto en la pila y hacer que el kernel restaure registros arbitrarios: RIP, RSP, RAX, etc. Esto permite invocar syscalls directamente (como execve("/bin/sh")) sin depender de gadgets complejos.

Para esta técnica necesitamos controlar bastante espacio en el stack, pero nosotros solo podemos sobrescribir el Saved RBP y el Saved Ret, ¿cómo podemos pasar de esto a poder hacer el SROP?

Stack Pivot

Un stack pivot es una técnica de explotación que consiste en cambiar el RSP a una zona de memoria controlada por el atacante. Esto es útil cuando el espacio disponible en el stack original es limitado y no permite construir una cadena ROP completa.

Para lograrlo, se utilizan gadgets como leave; ret, pop rsp; ret o similares, que permiten redirigir el stack hacia un buffer controlado (por ejemplo, en .bss). Una vez hecho el pivot, la ejecución continúa leyendo direcciones desde esa nueva ubicación, permitiendo ejecutar cadenas ROP más largas.

Estrategia de Explotación

Tenemos una parte perfecta a la que dirigir el stack pivot, ¡el .bss que controlamos! Pero primero habrá que leakear el PIE, ¿recordais que en la función login el input se leía con read? Esto es importante, porque read al leer no introduce un \x00, por lo que podemos conseguir un Out Of Bounds Read, ¡y justo después tenemos nuestra dirección de memoria de nuestro .bss input.

La estrategia de explotacion será la siguiente:

  1. Leakear el .bss input con login.
  2. Buffer Overflow en manage para hacer un Stack Pivot a la zona .bss controlada por nosotros.
  3. En la zona .bss usar el gadget pop rax; syscall; ret para hacer SROP y llamar a execve("/bin/sh").

El siguiente paso será preparar el sigcontext, yo lo haré con pwntools. En este deberemos poner en los siguientes registros:

  • RDI: un puntero a la cadena "/bin/sh"
  • RSI y RDX: 0
  • RAX: 59 (syscall de execve)
  • RIP: gadget syscall; ret
