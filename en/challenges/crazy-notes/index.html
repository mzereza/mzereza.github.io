Program Flow

This program shows a menu with the following options:

  1. Allocate a world struct of 0x18 bytes, and a buffer of size between 0x30 and 0x50 bytes.
  2. Edit a world struct and its buffer, if it’s within heap range.
  3. Show the data of a world struct and then choose whether to free the struct and the buffer. When freeing we erase all data from the world struct, but we don’t lose the reference to it (UAF!).

Exploitation Strategy

When I do heap challenges, my methodology is usually:

  1. Gain Arbitrary Read/Write or OOB Read/Write.
  2. Leak Libc.
  3. Find a way to gain Code Execution.

Gaining Arbitrary Read & Write

We only have 8 allocations, so we need to use them wisely. When creating a world, two malloc() calls are made: one of 0x20 bytes where we fully control the first 0xc bytes, and a second of a selectable size between 0x30 and 0x50 that we fully control. The idea is to allocate three worlds, free the first and show its data to leak the Heap Base, then free the second and do TCache Poisoning to allocate a chunk at the void* buffer of the second. This lets us control that pointer and gain Arbitrary Read & Write within the heap. The third one is important for later.

This challenge uses glibc version 2.39, which has Safe Linking.

Safe Linking

Safe Linking is a TCache security measure. What it does is store (heap_base >> 12) XOR next_ptr instead of next_ptr in the Forward Pointer of freed chunks. So to do TCache Poisoning we’ll first need to leak the Heap Base. This is quite easy with our UAF — just read the Forward Pointer of the first freed chunk, since it will be (heap_base >> 12) XOR 0.

def create(index, size, tierra, agua, aire, data):
    p.sendlineafter(b">", b"1")
    p.sendlineafter(b">",str(index).encode())
    p.sendlineafter(b">",str(size).encode())
    p.sendlineafter(b">",str(tierra).encode())
    p.sendlineafter(b">",str(agua).encode())
    p.sendlineafter(b">",str(aire).encode())
    p.sendlineafter(b">",data)

def edit(index, tierra, agua, aire, data):
    p.sendlineafter(b">",b"2")
    p.sendlineafter(b">",str(index).encode())
    p.sendlineafter(b">",str(tierra).encode())
    p.sendlineafter(b">",str(agua).encode())
    p.sendlineafter(b">",str(aire).encode())
    if(len(data)>0):
        p.sendafter(b">",data)

def delete(index, sure):
    p.sendlineafter(b">",b"3")
    p.sendlineafter(b">",str(index).encode())
    p.recvuntil(b"Stats : ")
    
    num1 = p.recvuntil(b" - ")
    num2 = p.recvuntil(b" - ")
    num3 = p.recvuntil(b"\n")

    num1 = int(re.sub(rb'\D', b'', num1))
    num2 = int(re.sub(rb'\D', b'', num2))
    num3 = int(re.sub(rb'\D', b'', num3))
    
    fd = num1+(num2<<32)
    p.sendlineafter(b">",sure)
    return fd

def get_first8(index):
    return delete(index, b"n")

def free(index):
    return delete(index, b"Y")


def rewrite_fd(index, fd, aire, data):
    tierra  = fd & 0xffffffff
    agua = (fd >> 32) & 0xffffffff
    edit(index, tierra, agua, aire, data)

def getworld(index):
    p.sendlineafter(b">", b"3")
    p.sendlineafter(b">",str(index).encode())
    p.recvuntil(b"World : ")
    data = p.recvline()
    p.sendlineafter(b">", b"n")
    return data

Now by modifying the first 8 bytes in chunk 0 we control the buffer of chunk 1. With this done, how do we leak Libc?

Leaking Libc

Normally in heap challenges, Libc is leaked by making a chunk fall into the unsorted bin, since when a chunk falls into the unsorted bin its fd and bk point to the main arena. But here we can’t allocate a chunk large enough for it to go to the unsorted bin when freed.

What we can do is create a fake chunk in the heap, and put its address in the buffer of a world, so that when the world is freed, our fake chunk is also freed and falls into the unsorted bin.

A few things to keep in mind here:

  1. It has to be larger than 0x410 so it doesn’t fall into the TCache. (I’ll make it 0x500)
  2. We have to create a fake guard chunk after it to prevent consolidation.
  3. For the heap to think the guard chunk is in use, we have to create another fake chunk after it with the PREV_INUSE bit set.

Gaining Code Execution

We have our libc leak — now what?

Once we’ve reached this point there are several ways to gain Code Execution. One of them is with the tls_dtor list. I’ll leave the challenge author’s writeup using that technique here: Writeup with TLS_dtor list

I’ll solve the challenge with FSOP. I’ll create a fake struct _IO_FILE_plus on the heap and put it in _IO_list_all so that when _IO_flush_all() is executed as we exit the program it calls system("/bin/sh").

FSOP (File Stream Oriented Programming)

FSOP is a glibc exploitation technique that consists of abusing the internal structures of file streams (FILE, used by functions like printf, fgets, etc.) to achieve code execution.

The idea is that glibc represents files using structures (_IO_FILE) that contain function pointers (vtable). If an attacker can corrupt one of these structures (for example, through a heap overflow or a Use After Free), they can redirect those pointers so that when the program invokes an operation on the stream, it ends up executing controlled code.

Overwriting the vtable

To gain arbitrary code execution we’d need to overwrite the vtable so that when _IO_flush_all() calls a function in our File Struct it actually calls system. Here’s a diagram so you can visualize it better:

source https://pwn.college/software-exploitation/file-struct-exploits/

But here we run into a problem: glibc has a protection that verifies the vtable is in the __libc_IO_vtables section.

More info at: https://elixir.bootlin.com/glibc/glibc-2.31/source/libio/libioP.h#L935

Bypassing the vtable protection

To bypass the protection we can overwrite the vtable within the __libc_IO_vtables section to call _IO_wfile_overflow. This function then calls _IO_wdoallocbuf which uses an unprotected vtable in FILE->wide_data.

By doing that and overwriting the vtable in wide_data we’ll be able to call system().

Now, how do we pass the "/bin/sh" string? This is relatively straightforward — when any function from the vtable is called, the pointer to the File Struct is passed as the first argument. We’d just need to put that string at the beginning of the File Struct, right?

Well, we can’t put "/bin/sh" directly at the start because the flags are at the beginning of the File Struct, and we’d corrupt them in a way that _IO_wfile_overflow wouldn’t be called. The solution is to use the string "\x3b\x01\x01\x01\x01\x01\x01\x01" — this string passes the flags checks, and since \x3b represents a ";", the following string will be executed as another command.

We also need to keep a few other things in mind:

  1. _IO_write_ptr must be greater than 0 and less than _IO_write_end
  2. _lock must be a writable memory address with value 0.

Now for the last step — overwriting _IO_list_all.

Overwriting _IO_list_all

To overwrite _IO_list_all we can’t use the same Tcache poisoning method as before, because we’d exceed the maximum number of worlds we can create.

Since we have Arbitrary Write within the heap, we can overwrite the Tcache Entry and modify the entry for 0x20 chunks so the next one returned is _IO_list_all. Once done, simply write the pointer to our fake File Struct and exit the program.
