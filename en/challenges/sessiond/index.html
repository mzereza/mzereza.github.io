Program Flow:

The program shows us a menu with different functions:

The manage function asks us for a size to enter data, then reads our data into a segment of the .BSS and copies it to a variable on the stack. There is a buffer overflow here, since we can write 0x210 bytes but the buffer is 0x200 bytes. We can overwrite the Saved RBP and the Saved return address!

The login function puts the memory address of input at offset 0x4070 and asks us for a username, which it reads with read at offset 0x4060. This will be important for later, and then displays it on screen.

The status function looks strange in Ghidra’s pseudocode, so let’s look at the assembly directly:

In the assembly we can see it prints a string with the write syscall and does a JMP to the return, but in between there’s a very powerful gadget: POP RAX; SYSCALL; RET. This gadget lets us execute whatever syscall we want. We can’t reach this gadget conventionally, but if we control RIP we can jump to that instruction.

The question here is: How can we go from this to arbitrary command execution?

We can overwrite the Saved RIP with the address of the gadget, but we have no control over the registers or the rest of the stack. Also, what can we do with just one syscall?

Sigreturn Oriented Programming (SROP)

SROP is an exploitation technique that leverages the sigreturn syscall to fully control the CPU register state. On Linux systems, sigreturn is used internally to restore a process’s context after handling a signal, loading register values from a structure on the stack (sigcontext).

The key is that, if an attacker can force execution of sigreturn (for example, through a gadget or a syscall), they can manually construct that context on the stack and make the kernel restore arbitrary registers: RIP, RSP, RAX, etc. This allows invoking syscalls directly (such as execve("/bin/sh")) without relying on complex gadgets.

For this technique we need to control quite a bit of space on the stack, but we can only overwrite the Saved RBP and the Saved Ret. How can we go from this to being able to do SROP?

Stack Pivot

A stack pivot is an exploitation technique that consists of changing the RSP to a memory zone controlled by the attacker. This is useful when the available space on the original stack is limited and doesn’t allow building a complete ROP chain.

To achieve this, gadgets like leave; ret, pop rsp; ret or similar are used, which allow redirecting the stack toward a controlled buffer (for example, in .bss). Once the pivot is done, execution continues reading addresses from that new location, allowing longer ROP chains to be executed.

Exploitation Strategy

We have a perfect place to direct the stack pivot — the .bss we control! But first we’ll need to leak the PIE. Remember that in the login function the input was read with read? This is important, because read doesn’t insert a \x00 when reading, so we can get an Out Of Bounds Read — and right after we have our memory address of our .bss input.

The exploitation strategy will be:

  1. Leak the .bss input with login.
  2. Buffer Overflow in manage to do a Stack Pivot to the .bss zone we control.
  3. In the .bss zone, use the gadget pop rax; syscall; ret to do SROP and call execve("/bin/sh").

For this we’ll use the login function and fill the entire input buffer. This will let us leak our controlled zone in the .bss.

binary = "./sessiond"

The next step is to prepare the sigcontext. I’ll do it with pwntools. We need to set the following registers:

  • RDI: a pointer to the string "/bin/sh"
  • RSI and RDX: 0
  • RAX: 59 (execve syscall)
  • RIP: gadget syscall; ret

The pointer to the /bin/sh string can be obtained by putting that string as the username.

Now we just need to build our payload, which follows this structure:

  1. 0 (Fake RBP)
  2. gadget pop rax; ret
  3. 15 (sigreturn syscall)
  4. sigcontext
  5. Padding up to Saved RBP
  6. Address of our .bss input we’re pivoting to (Saved RBP)
  7. gadget leave; ret (Saved Return Address)
