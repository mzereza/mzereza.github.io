RetSpill: From Kernel CHF to LPE

April 02, 2026

Disclaimer

This technique is not my own; you can read the original research here:
https://adamdoupe.com/publications/retspill-ccs2023.pdf

TL;DR

RetSpill allows transforming a limited kernel control flow hijack into a more powerful primitive by taking advantage of how the kernel manages the stack.

In this post I explain the intuition, how I understood it by solving a challenge, and how it translates into an LPE.

Introduction

RetSpill is a technique that appears in the context of kernel exploitation when we have a Control Flow Hijack (CHF) but not enough control to do ROP directly.

I learned this technique while solving a PwnCollege challenge where I had already achieved a CFH, but didn’t know what to do with that limited control.

In this post I won’t repeat the paper, but instead explain:

  • What problem RetSpill solves
  • The intuition behind the technique
  • How it’s applied in a real scenario

The Problem: CHF Is Not Always Enough

Before talking about RetSpill, we need to understand the key limitation.

In many kernel challenges:

  • You can redirect execution (CHF)
  • But you don’t fully control the registers or the stack
  • Or you can’t find a gadget to pivot

On top of that, you’re usually restricted by mitigations (SMEP, SMAP, KASLR, etc.)

In short: you have control… but not enough to achieve a complete exploit.

This is where RetSpill comes in.

RetSpill Intuition

The core idea of RetSpill is:

Take advantage of how the kernel saves and reuses user-controlled data on the stack so that, for example, with a gadget like add rsp, X ; ret you can pivot and execute a complete ROP Chain.

Explained informally:

  • The kernel constantly pushes/pops values on the stack
  • Some of those values may end up being indirectly controllable
  • If you can align that with a ret, you can influence the execution flow

It’s not a classic stack overflow, but rather: reuse of residual stack state.

What Gets “Spilled” Exactly?

In this post I’ll focus specifically on abusing the pt_regs structure. For more information about other ways to carry out RetSpill, see the original paper.

When a syscall is made, the kernel transitions from userland to kernel space. Before executing any logic, it needs to preserve the process’s userland context.

To do this, it saves the register state on the current thread’s kernel stack, packaging it in a structure called pt_regs.

This structure contains information such as:

  • the instruction pointer (rip)
  • the stack pointer (rsp)
  • flags (rflags)
  • general-purpose registers (rax, rbx, etc.)

In many exploitation scenarios, several of these registers come directly from userland, so we have some degree of control over their content.

This opens up an interesting possibility: we can make values we control end up inside the stack region where pt_regs is built. From there, if we can redirect execution to a suitable gadget (for example, add rsp, X ; ret), we can pivot the stack to that area and reuse those values as a ROP chain.

In other words, pt_regs stops being just a context-saving mechanism and becomes a useful surface for building execution control.

Limitations and Conditions

RetSpill is not an attack on its own, but rather a technique that builds on certain prior primitives. In particular, you usually need:

  • Some kind of Control Flow Hijack (CHF)
  • A KASLR leak, if that mitigation is active

That is, RetSpill doesn’t give you initial control, but rather serves to amplify already existing but limited control.

Furthermore, in many real-world scenarios there’s an important trade-off: the very bug that gives you the CHF may restrict which registers you control.

For example, you might manage to redirect execution, but:

  • you don’t fully control the registers in Userland
  • some registers have been overwritten or sanitized
  • the stack state is not entirely predictable

These limitations can complicate the use of RetSpill, since the technique depends precisely on leveraging existing state on the stack. If that state is not sufficiently controllable or reliable, exploitation becomes much more complex.

Conclusion

RetSpill is not a technique to gain control, but to amplify limited control.

And that’s precisely where its value lies: in modern kernel exploitation scenarios, where new mitigations make it increasingly harder to achieve a complete exploit directly, techniques like this allow squeezing the maximum out of weak or incomplete primitives.
